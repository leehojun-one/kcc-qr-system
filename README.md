# KCC 홈씨씨 창호 QR·바코드 관제 시스템 (프로토타입)

발주서 바코드 + 거실 메인창 QR을 현장에서 매칭해, 한 집의 창호 데이터를 본사 DB에 1:1로 묶고
→ **사고/AS를 생애주기 단계별로 접수·연동**하는 시스템.

**AS 신고 3종 (단계별 진입점/라우팅)**
| 유형 | 시점 | 신고자 | 1차 수신 | 비고 |
|---|---|---|---|---|
| 당일사고 | 설치 당일 | 시공팀 | 가공처 + 관제 | 99% 가공처 액션, 긴급 |
| 입주전 AS | 인테리어 공사중·입주 전 | 대리점/현장 | 관제 + 영업 | 가공처는 관제가 검토 후 지시 |
| 사용중 AS | 입주 후 | 고객(정품등록) | 본사 관제 | 보증 AS 배정 |

당일사고·입주전AS는 하자(입주 전 발견)로 **하자율에 집계**, 사용중 AS는 보증건으로 별도 집계.

경영진 보고 → 본사 IT 서버 이관을 전제로 한 **작동하는 프로토타입**입니다.

---

## 실행 방법

```bash
pip install -r requirements.txt
streamlit run app.py
```

브라우저가 열리면 좌측에서 3개 화면을 전환합니다. 최초 실행 시 데모 데이터가 자동 적재됩니다.
(데이터 초기화: `kcc_qr.db` 파일 삭제 후 재실행)

## 4개 화면 (사용자별 분리)

> ②파트너사·③고객 앱은 **QR을 찍으면 그 현장으로 바로 열림**. 거실창 QR이 `…streamlit.app/?qr=일련번호`
> 를 담고 있어, 스캔 시 랜딩에서 **[입주민] / [현장 관리자(파트너사)]** 만 고르면 됨.
> (배포 후 Streamlit Secrets 에 `base_url` 을 넣어야 QR이 실제 앱 주소로 연결됨.)

### ① 시공팀 앱  (내부 · 설치 당일)
- **현장 등록**: 발주서 바코드(창호견적번호) + 거실 메인창 QR 일련번호 연결.
  현장정보(단가 제외)와 창호 18개 목록 표시. `QR 연결` 누르면 그 집 데이터가 QR에 입혀짐.
- **당일사고 등록**: 문제 창 → 1차 귀책 → 사진 → 접수 → **가공처+관제 동시 발송**.
  (시공팀은 당일 시공 후 현장을 떠나므로 여기까지만 담당)

### ② 파트너사 앱  (인테리어 파트너 · 고객 인계 전)
- 공사중 현장을 관리하는 **대표/실장/현장소장**이 **입주전 AS** 접수.
- 문제 창 → 신고자 → 귀책(운반·시공 파손/타공정 훼손 포함) → 사진 → **관제+영업만 발송**.
  (가공처는 관제가 검토 후 지시)

### ③ 고객 앱  (외부 · 입주 후 · 내부정보 비노출)
- 거실 메인창 QR 스캔 → **첫 스캔=정품등록**(연락처+입주일), **이후=A/S 접수**.
- A/S: 설치위치 체크 + 증상 체크 + 직접입력. 시공팀/가공처/영업자 정보는 안 보임.

### ④ 관제센터  (본사)
- KPI: 누적 사고/AS, **하자율(확정 기준)**, 누적 시공 창호, 미확인 건 (+당일/입주전 분할).
- 귀책별 분포 / **가공처별 하자율**(파트너 관리용).
- **2차 귀책 확정 큐**: 1차 잠정귀책 → 관제가 검토 후 확정. *확정 건만 통계에 반영*.
  입주전AS 건은 **`가공처에 지시 발송`** 버튼으로 관제가 가공처 액션을 지시.
- 최근 티켓(유형/신고자/에스컬레이션) + 고객 사용중 AS 현황.
### ⑤ 관리자 (발행 · 본사 PIN)
- **발주서 업로드 → 바코드 발행**: 시공발주서 xlsx 올리면 파싱·현장등록 + 바코드 PNG 다운로드.
  (시공팀은 견적 권한이 없으므로 본사가 발행 → 견적프로그램 연동 전 임시 다리)
- **QR 100장 일괄 발행**: 빈 고유번호 QR을 A4 PDF로 출력해 미리 인쇄.

### ⑥ 가공처 (드롭다운 선택)
- 가공처 선택 → 내게 접수된 사고 알람·목록 → 접수확인/처리예정/완료 입력.

### ⑦ 영업사원 (드롭다운 선택)
- 영업사원 선택 → 내 담당 현장의 사고/AS 열람 (상황 파악 전용, 처리는 가공처·관제).

> ④관제·⑤관리자는 **PIN 잠금**(Secrets의 control_pin/admin_pin · 로컬 기본 0000).

## 파일 구조

| 파일 | 역할 |
|---|---|
| `app.py` | Streamlit 메인 (3개 화면) |
| `parser.py` | 시공발주서(.xlsx) → 현장정보+창호 추출 (단가 제외) |
| `db.py` | SQLite 백엔드 (시트 5종) + 데모 시드 + 하자율 기준선 |
| `barcode_qr.py` | Code128 바코드 / QR 생성 |
| `notify.py` | 알림 발송(데모 시뮬레이션) + 에스컬레이션 로직 |
| `seed_site.json` | 실제 발주서 1현장 데이터(파싱 결과) |

## 본사 IT 서버 이관 가이드

설계상 교체 지점이 3곳으로 분리돼 있습니다.

1. **DB → 사내 서버**: `db.py`의 함수 시그니처(get_site, add_incident 등)를 유지한 채
   내부를 사내 DB(MySQL 등) 또는 구글시트 API로 교체. `app.py`는 수정 불필요.
2. **알림 실발송**: `notify.py`의 `_send_via_provider()` 에 솔라피/알리고 등
   카카오 알림톡 딜러사 API 연결. 알림톡 템플릿 사전심사(2~10일) 필요, 실패 시 SMS 대체발송 설정.
   `dispatch(..., simulate=False)` 로 전환.
3. **하자율 분모**: `db.py`의 `VENDOR_BASELINE`(데모용 누적 시공량)을
   실제 sites/windows 집계 또는 ERP 시공 실적 집계로 교체.

### 바코드/QR 운영
- 발주서 '오더번호' 칸 = 창호견적번호 Code128 바코드.
- 거실 메인창 13년 보증스티커 = 고유 일련번호 QR(미리 인쇄 권장).
- 현장에서 두 개를 스캔해 1:1 매칭.

### 실측 데이터 연동
`parser.py` 는 ERP에서 출력되는 모든 '출고/시공 발주서' xlsx에 사용 가능:
```bash
python parser.py /path/to/시공발주서.xlsx
```

## 경영진 데모 시나리오 (생애주기 순서, 약 7분)

1. **① 시공팀 앱 ▸ 현장 등록**: 방배 임광 현장 선택 → 바코드/QR 보여주고 `✅ QR 연결`.
2. **① 당일사고 등록**: 거실발코니 / 공장오제작 / 사진 → `사고 접수` →
   **가공처+관제 동시 발송** 메시지 확인.
3. **④ 관제센터**: 방금 건이 미확인/확정대기 큐에 뜬 것 확인 → `확정` →
   하자율·귀책 통계에 반영되는 것 확인.
4. **② 파트너사 앱**: 같은 현장 → 신고자(실장) / 타공정 훼손 → `입주전 AS 접수` →
   **관제+영업만** 발송(가공처 빠짐) 확인.
5. **④ 관제센터**: 입주전AS 건에서 `📤 가공처에 지시 발송` → 관제가 가공처로 내려보냄.
6. **③ 고객 앱**: 같은 현장 QR → 정품등록(연락처) → 재진입 → A/S 접수(내부정보 없음) →
   ④ 관제센터 하단 사용중 A/S 현황에 표시.

> **폰 시연**: 배포 후 Secrets에 `base_url`을 넣으면 ①에서 만든 QR을 폰 카메라로 찍어
> 4·6번을 실제 스캔으로 시연 가능. (랜딩에서 입주민/파트너사 선택)
